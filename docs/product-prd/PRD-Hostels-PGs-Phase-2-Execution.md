## 1. Objective & Scope

Phase 2 focuses on transforming the Hostels & PGs experience into a scalable, search-optimised accommodation directory with 90+ verified listings. The scope covers:

- Completing the outstanding feature backlog called out in January 2025 deployment notes.
- Building a reliable ingestion workflow to transform external CSV data into our MySQL schema.
- Strengthening SEO/SEM so each hostel/PG earns individual search visibility on Google and Bing.
- Aligning UX copy and data models with hostel/PG terminology (avoid “property” confusion).
- Preparing owner-side tooling to nurture leads generated by the upcoming listing expansion.

Execution of this PRD will be sequenced; no production changes are triggered until workstreams are scheduled and approved.

## 3. Data Ingestion Plan (90+ Listings)

### 3.1 Source Overview
- CSV file: `First-Set-OMR-Hostel-PG.csv`
- Mixed accommodation dataset (hotels, PGs, coliving) with Bing Places IDs, contact data, social links.
- Structure differs from MyOMR schema (`hostels_pgs`, `property_photos`, `property_owners`, etc.).

### 3.2 Mapping Strategy

| Target Table.Column | Required? | Source Field | Transformation Notes |
| --- | --- | --- | --- |
| `property_owners.full_name` | ✅ | `Name` (organisation) | Create owner profiles grouped by unique email/phone domains. |
| `property_owners.email` | ⚠️ | `Emails` | Extract primary; fallback to `info@hostel.com`. Deduplicate case-insensitive. |
| `property_owners.phone` | ⚠️ | `Phone` | Normalise with `+91` prefix; if multiple, pick primary. |
| `hostels_pgs.property_name` | ✅ | `Name` | Title case; ensure hostel/PG suffix retained. |
| `hostels_pgs.property_type` | ✅ | Derived | Map keywords: “Hotel/Resort” → Hostel (if multi-sharing), else flag for review. Use manual classification checklist. |
| `hostels_pgs.address` | ✅ | `Address` | Append locality + state; validate length. |
| `hostels_pgs.locality` | ✅ | Derived from `Address` | Use regex/lookup (Navalur, Sholinganallur, etc.). Unknowns flagged for manual QA. |
| `hostels_pgs.latitude`, `longitude` | Optional | `Latitude`, `Longitude` | Convert to decimal; blank if missing. |
| `hostels_pgs.featured_image` | Optional | `Featured image` | Validate URL; plan to download + host once media pipeline ready. |
| `hostels_pgs.brief_overview` | ✅ | Generated summary | Auto-generate 2 sentence summary using address + category; manual review queue. |
| `hostels_pgs.full_description` | ✅ | Content template | Compose from data points (amenities, rating). Use content writer pass for top listings. |
| `hostels_pgs.facilities` | ✅ | Derived from category + manual tagging | Build amenity taxonomy (`WiFi`, `Meals`, `AC`, etc.) using heuristics. |
| `hostels_pgs.food_options` | Optional | Manual | Default “Veg & Non-Veg available” unless known; review needed. |
| `hostels_pgs.gender_preference` | ✅ | Manual classification | Use brand cues / additional research. Unknown defaults to `Co-living` with review flag. |
| `hostels_pgs.contact_phone` | ⚠️ | `Phone` | Same as owner unless property-specific numbers found. |
| `hostels_pgs.contact_email` | ⚠️ | `Emails` | Use property-level email if distinct. |
| `hostels_pgs.slug` | ✅ | Generated | `slugify(property_name + locality)`; enforce uniqueness. |
| `property_photos.photo_url` | Optional | `Featured image` | Store raw URL initially; schedule migration to local storage. |
| `property_inquiries` | N/A | - | No import; future data only. |

### 3.3 Workflow
1. **Data Audit:** Identify non-hostel entries (e.g., chain hotels) for exclusion or separate category.
2. **Normalization Script:** Build one-off PHP/CLI script to parse CSV → canonical JSON → SQL inserts.
3. **Owner Assignment:** Create or reuse owners; log mapping file for traceability.
4. **Slug & Duplicate Guard:** Check existing database to avoid duplicate listings.
5. **Review Queue:** Flag uncertain fields (gender preference, food, amenities) for manual validation.
6. **Staging Import:** Load into staging DB, run QA (filter performance, detail page rendering).
7. **Approval:** Business sign-off before pushing to production.

### 3.4 Deliverables
- Mapping document (this table).
- Import script with dry-run mode + rollback.
- QA checklist for data verification.
- Update to documentation describing ingestion process.

## 4. Terminology Alignment Plan

To prevent confusion with real-estate sales, all user-facing language must emphasise accommodation:

- Replace “Property/Properties” with “Hostel” / “PG” / “Accommodation” where contextually correct (labels, headings, buttons, breadcrumbs).
- Update helpers (e.g., `property-functions.php`) naming only when safe; ensure backwards compatibility via alias functions if needed.
- Revise structured data types: use `LodgingBusiness`, `Hostel`, or `Residence` taxonomy where appropriate.
- Update sitemap sections and canonical URLs to reinforce hostel/PG keywords.

Deliverable: UI copy audit checklist + diff-ready string table.

## 5. SEO & SEM Strengthening

### 5.1 Per-Listing On-Page Strategy
- **Unique Titles:** `{{Hostel Name}} | {{Locality}} Hostels & PGs in OMR Chennai | MyOMR`.
- **Meta Descriptions:** 150–160 char summaries including gender preference, price range, locality.
- **Structured Data:** Extend `generatePropertySchema()` to include `@type` overrides (Hostel/Residence) and `geo`, `amenityFeature`, `aggregateRating` (when available).
- **Header Hierarchy:** Ensure `H1` uses hostel name, `H2` for location, facilities, pricing.
- **Keyword Density:** Incorporate phrases like “hostel in {{locality}}”, “PG accommodation near {{landmark}}”.
- **Image SEO:** Alt text describing room types/amenities; host images locally with descriptive filenames.

### 5.2 Technical SEO
- Update sitemap generator to include every hostel detail URL weekly.
- Add `lastmod` to reflect meaningful content changes.
- Configure canonical URLs after slug creation; ensure 1:1 with detail pages.
- Create internal linking widgets: “Nearby Hostels,” “Similar PGs for {{Gender}}”.
- Generate HTML landing pages by locality/gender (e.g., `/omr-hostels-pgs/navalur-girls/`).

### 5.3 Off-Page & Search Console
- Submit updated sitemap to Google & Bing Webmaster Tools upon import.
- Track queries in Search Console; build dashboards for top hostel keywords.
- Encourage owners to link back to their listing pages (referral backlinks).

### 5.4 Bing-Specific Actions
- Use Bing Webmaster Tools URL submission API post-launch.
- Leverage `BingPlaces` attributes from CSV to cross-link listings when appropriate.
- Monitor Bing index coverage separately due to dataset origin.

### 5.5 SEM Readiness
- Prepare structured landing pages for paid campaigns (future).
- Define UTM taxonomy (`utm_source=sem`, etc.) for analytics consistency.

## 8. Risks & Mitigations

- **Data Quality Risk:** CSV contains hotels not suitable for PG category → implement manual review + tagging guidelines.
- **Performance Risk:** Current PHP filtering may not scale to 90+ listings → prioritise SQL filter refactor before import.
- **Security Risk:** Email-only owner login allows spoofing → integrate verified login before enabling owner dashboard enhancements.
- **SEO Risk:** Duplicate or thin content lowers ranking → ensure bespoke copy + alt text per listing; avoid mass-produced boilerplate.
- **Operational Risk:** Featured auto-expiry requires cron access on cPanel → coordinate with ops for scheduling.
